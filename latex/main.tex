    Este trabajo presenta el diseño e implementación de un sistema de detección y clasificación de ciberataques en tiempo real mediante técnicas de Deep Learning y aprendizaje supervisado. El proyecto se basa en el análisis del tráfico de red de un servidor web, empleando un Autoencoder como mecanismo principal para la identificación de anomalías. Este modelo utiliza el error de reconstrucción generado sobre flujos de red para distinguir entre comportamientos benignos y maliciosos. Para el entrenamiento y validación del sistema, se ha utilizado el conjunto de datos de referencia CICIDS2017, complementado con capturas de tráfico personalizadas obtenidas en un entorno de laboratorio controlado.

Durante la fase de desarrollo, se ejecutaron múltiples auditorías de seguridad mediante herramientas profesionales como SQLMap y Hydra. Estas pruebas permitieron validar el rendimiento del sistema frente a escenarios de explotación reales, facilitando un análisis crítico sobre la capacidad de generalización del modelo y la mitigación de problemas relacionados con el sobreajuste en entornos automatizados.

    \vspace{5mm}

    Finalmente, el proyecto integra un sistema de monitorización y respuesta ante incidentes mediante la API de Telegram. Esta interfaz permite notificar al administrador de red de forma instantánea y facilita la aplicación de contramedidas defensivas en tiempo real. La arquitectura propuesta combina con éxito la detección de anomalías, la clasificación jerárquica de amenazas y la capacidad de respuesta interactiva, explorando el potencial de las técnicas actuales de inteligencia artificial aplicadas a la protección de infraestructuras críticas en el ámbito de la ciberseguridad.

    \section{Introducción}\label{sec: introduccion}

    \vspace{5mm}

    En el actual ecosistema digital, las páginas web son uno de los principales servicios más relevantes, ya que son usados continuamente por miles de usuarios en momentos determinados del día, suponiendo un gran riesgo a nivel de seguridad. Los ciberataques hoy en día están en auge, evolucionando en cuanto a complejidad y automatización, y esto hace que los sistemas tradicionales de detección de intrusiones (IDS) basados en patrones muy concretos se queden obsoletos, ya que se ven muy limitados a identificar ataques con fluctuaciones muy leves que no han sido previamente identificados.

    \vspace{5mm}

    Por ello, este trabajo surge a partir de dicho problema, ante la necesidad de monitorizar en tiempo real flujos de red que llegan a un servicio web en concreto, distinguiendo lo mejor posible el tráfico benigno y el malicioso. Debido al volumen de datos que esto presenta, no es posible proponer una solución manual a este problema, y por ello se busca una solución basada en inteligencia artificial y modelos predictivos, aprendiendo la estructura intrínsica de los datos y reaccionando de manera proactiva ante desviaciones que puedan suponer una amenaza para un servicio.

    \vspace{5mm}

    \subsection{Solución Propuesta: Arquitectura Híbrida de Detección y Respuesta}

    En este proyecto se propone una arquitectura que combina el aprendizaje no supervisado para detectar flujos de red maliciosos, para posteriormente proceder con la clasificación de la amenaza mediante aprendizaje supervisado, siendo el objetivo principal de este trabajo la transformación de datos de red masivos y en crudo a un formato mucho más legible para un administrador de seguridad que se encarga de monitorizar el servicio. En definitiva, este trabajo se constituye en cuatro etapas críticas:

    \vspace{5mm}

    \begin{enumerate}
        \item \textbf{Ingesta de datos y extracción de características:} Con el apoyo de la librería \texttt{nfstream} \cite{nfstream_project} se transforma el tráfico de red en un flujo de datos, extrayendo variables muy significativas para los modelos de predicción, como el número de paquetes enviados y recibidos, métricas relacionadas con el tamaño de los paquetes o banderas TCP.
    
        \vspace{5mm}

        \item \textbf{Detección de anomalías mediante Autoencoders:} Se destaca el uso de una red neuronal de tipo \textit{Autoencoder} entrenada únicamente con flujos benignos, pretendiendo recrear cada flujo de datos recibido por el servidor, de manera que si dicho flujo presenta un error de reconstrucción (\textit{MSE}) alto, se confirma la presencia de tráfico malicioso, ya que la red no reconoce los patrones de dicho flujo al no ser legítimo (ya que fue entrenada con datos no maliciosos) disparando el error de reconstrucción y actuando como una señal de alerta ante una posible amenaza.
        
        \vspace{5mm}

        \item \textbf{Clasificación mediante Random Forest:} Una vez hemos detectado la anomalía, procedemos a su clasificación mediante un \textit{Random Forest} previamente entrenado con datos maliciosos, al contrario que la red neuronal. Esto permite clasificar el tipo de ataque recibido y dar una vista mucho más específica de la naturaleza del ataque
        
        \vspace{5mm}
        
        \item \textbf{Intervención del administrador: } Por último, nuestro sistema incluye una interfaz de respuesta mediante la API de Telegram, notificando la anomalía y abriendo la posibilidad de un bloqueo inmediato de la IP del atacante para proteger la integridad del servicio.
    \end{enumerate}

    \subsection{Metodología de Validación y Entorno de Laboratorio}

    \vspace{5mm}

    En cuanto al entorno del sistema se ha simulado un escenario de red completamente real, mediante un servidor web que sigue una arquitectura LAMP (\textit{Linux, Apache, MySQL, PHP}), el cual ha sido sometido a diferentes ataques y auditorías de seguridad para garantizar la efectivadad de la solución propuesta y un sistema de defensa activa significantemente robusto. Para generar los distintos flujos de red maliciosos hacia el servidor, se han empleado las siguientes herramientas, las cuales forman parte de cualquier set de herramientas de pentesting:

    \vspace{5mm}

    \begin{itemize}
        \item \textbf{SQLMap:} Como su propio nombre indica, se centra en ataques de inyección SQL, lo que permite al sistema detectar cualquier intento de volcado de la base de datos con el fin, en la mayoría de casos, de obtención de credenciales.
        
        \vspace{5mm}

        \item \textbf{Hydra:} Esta herramienta es muy popular ya que permite simular ataques de fuerza bruta a través de un diccionario, lo que genera una gran cantidad de flujos de escasa duración y tamaño de paquetes, un ataque clásico pero muy eficaz y de los más habituales en la actualidad.
        
        \vspace{5mm}

        \item \textbf{Hulk/GoldenEye:} Empleados para la simulación de ataques de Denegación de Servicio (DoS), ya que saturan el propio servidor a través de peticiones HTTP. De esta manera se puede comprobar la respuesta del IDS bajo una alta situación de estrés.
    \end{itemize}

    \vspace{5mm}

    En definitiva, el uso del conjunto de datos \texttt{CICIDS2017} constituye uno de los pilares principales de este proyecto, ya que forma la estructura necesaria de aprendizaje de cada uno de los modelos para aprender patrones de ataques estandarizados, además de capturas locales que se han debido implementar para una mayor adaptación a la infraestructura de nuestro sistema, lo que reduce la distancia entre la teoría propuesta y su correspondiente implementación en un servidor en producción.

    \section{Marco Teórico}

    \vspace{5mm}



    \subsection{Detección de Anomalías}\label{subsec: deteccion_anomalias}

    \vspace{5mm}

    En esta sección vamos a explicar una parte vital de este proyecto, la cual consiste en la \textbf{detección de anomalías}. En el contexto de nuestro servidor web, una anomalía es un comportamiento extraño en el tráfico de red del servidor, sobre el cual nos vamos a centrar en aquellas peticiones de red fuera de lo común que se envían al servidor.

    \vspace{5mm}

    Más adelante vamos a saber diferenciar el tipo de ciberataque que se está intentando realizar por parte del atacante, pero en esta primera parte de la detección tenemos que diferenciar peticiones de red con un comportamiento benigno de uno maligno, y para ello vamos a utilizar un \textbf{autoencoder}.

    \vspace{5mm}



    \subsection{Autoencoder}\label{subsec: autoencoder}

    \vspace{5mm}

    Un \textbf{autoencoder} es una red neuronal que intenta replicar los datos recibidos de su capa de entrada $x$ en la capa de salida $r$. No obstante, en esta copia no nos vamos a fijar al utilizar cualquier \textit{autoencoder} en general, ya que nos interesa conocer el \textbf{error} que la red comete al intentar replicar los datos en la capa de salida $r$. De esta manera, si obtenemos un valor de error alto, los datos introducidos no se asemeja a los datos de entrenamiento de la red, y si obtenemos un valor de error bajo, lo contrario. Precisamente esto es en lo que nos vamos a basar para detectar logs malignos en el servidor, ya que si entrenamos la red con un set de datos benignos y comparamos el error producido por otras entradas benignas y malignas, el error será mucho mayor en los conjuntos de datos que el modelo no conoce, es decir, los datos malignos. 

    \vspace{5mm}

    Según se especifica en \cite{goodfellow2016deep}, este tipo de redes se componen en dos partes: El primer componente se denomina \textbf{encoder} $f$ (transforma lo datos de entrada $x$ en $h$), el cual identifica las variables más significativas y descarta el ruido innecesario, y un \textbf{decoder} $g$, que obtiene la reconstrucción del modelo $r$ a partir de $h$. Esta es la estructura básica de la red y se puede ver reflejada en la siguiente imagen:

    \vspace{5mm}

    \begin{figure}[H]
        \centering
        \includegraphics[width=0.40\textwidth]{../img/autoencoder_structure.png}\\
        \caption{Estructura elemental de un Autoencoder. Obtenida a partir de \cite{goodfellow2016deep}}
        \label{fig: autoencoder_structure}
    \end{figure}

    \vspace{5mm}

    Donde las funciones $f$ y $g$ serán definidas en la sección de implementación (\ref{subsec: autoencoder}). Una vez se ha visto la estructura básica de un \textit{autoencoder}, nos vamos a adentrar más específicamente en la red que se va a implementrar de manera práctica en este proyecto.

    \vspace{10mm}



    \subsection{Autoencoder Subcompleto}\label{subsec: autoencoder_subcompleto}

    \vspace{5mm}

    Se ha visto en la sección anterior que nuestro modelo tiene que aprender de sus datos de entrenamiento, y esto se puede lograr con un \textbf{autoencoder subcompleto}. La diferencia con respecto a un autoencoder genérico es que se van a limitar los datos obtenidos en $h$, obligando a que tengan una dimensión inferior que $x$. De esta manera $h$ es forzada a aprender las propiedades más representativas de los datos de entrenamiento, lo cual resultará en una reconstrucción final $r$ con ciertas discrepancias con respecto a los datos de entrada $x$. 

    \vspace{5mm}

    Podemos medir dichas discrepancias minimizando una función, la función de pérdida del modelo:

    \vspace{2.5mm}

    \begin{equation}
        L(x, g(f(x)))
        \label{eq: loss_function}
    \end{equation}

    \vspace{2.5mm}

    Donde $L$ es la función de pérdida que identifica si la reconstrucción del modelo $r = g(f(x))$ obtiene resultados similares a $x$. El entrenamiento de la red se basa en minimizar la función de pérdida que acabamos de ver (\ref{eq: loss_function}), y la función de pérdida que se va a usar en este trabajo es el \textbf{error cuadrático medio} \textit{MSE}, ya que penaliza los errores en mayor medida al elevar al cuadrado, es fácil de optimizar al ser continua y derivable en todo punto del espacio (esto facilita el entrenamiento de la red), representa la distancia euclídea por lo que facilita la representación, etc.

    \vspace{2.5mm}

    \begin{equation}
        MSE = \frac{1}{n} \sum_{i=0}^{n} \left( x_i - r_i \right)^{2}
        \label{eq: mse}
    \end{equation}

    \vspace{2.5mm}

    Siendo $x$ los datos de entrada, $r$ los valores de la reconstrucción, y $n$ el número total de datos en la capa de entrada y salida del modelo.

    \vspace{10mm}

    \subsection{Random Forest}\label{subsec: random_forest}

    \vspace{5mm}

    A continuación se va a ver detenidamente el funcionamiento teórico del modelo de clasificación utilizado en la implementación de este trabajo, el Random Forest. Este modelo fue escogido favorablemente en base a una comparación de resultados con otros modelos propuestos, debido a su mejor desempeño. Este apartado presenta un enfoque teórico - práctico, volcando la teoría a la práctica y su implementación en código.

    \vspace{5mm}

    \subsubsection{El estimador y el problema de la varianza}\label{subsubsec: estimador_problema_varianza}

    \vspace{5mm}

    Este modelo se compone de un boque de árboles, es decir, un conjunto de árboles de decisión. Un árbol de decisión es un modelo no paramétricos que segmenta el espacio de características mediante una serie de reglas que se van propagando hasta la decisión final. Por ello, destacan por su gran interpretabilidad y capacidad de capturar relaciones complejas entre las distintas variables de los datos, pero presentan un grave problema, la alta varianza. 

    \vspace{5mm}

    Imaginemos que un árbol de decisión contiene un sesgo ($bias$) reducido ya que es capaz de adaptarse con mucha precisión a la estructura del conjunto de datos de su entrenamiento. Esta sesgo reducido introduce una gran flexibilidad que lo convierte en un estimador inestable, ya que cualquier pequeña fluctuación en los datos de entrenamiento puede suponer un drástico cambio en la estructura de ramas y nodos del propio árbol, y esto es lo que se conoce como alta varianza. Entonces, si se tiene un conjunto de datos de entrenamiento $\mathbf{Z} = \{(x_1, y_1), \dots, (x_N, y_N)\}$, el algoritmo busca minimizar la impureza en cada partición, lo que suele conducir al sobreajuste ($overfitting$), comprometiendo la capacidad del modelo para generalizar datos distintos a los de su entrenamiento, ya que se memorizan dichos datos, perdiendo por completo la capacidad de predeicr correctamente cualquier otro dato.

    \vspace{5mm}

    \subsubsection{Bootstrap Aggregating (Bagging)}

    \vspace{5mm}

    Para aumentar la robustez de los árboles sin incrementar el sesgo, se emplea un método conocido como Bagging (\textit{Bootstrap Aggregating}), el cual se basa en que el promedio de múltiples estimadores independientes reduce la varianza global en el sistema de predicción. Si se dispusiera de $B$ conjuntos de entrenamiento independientes en un escenario ideal, cada uno con una varianza $\sigma^2$, la varianza del promedio sería $\sigma^2/B$. Sin embargo, en la práctica es imposible obtener múltiples conjuntos independientes, por lo que el Bagging simula esta condición mediante otro proceso conocido como \textbf{Bootstrap}.

    \vspace{5mm}

    Esta técnica consiste en generar $B$ muestras distintas $\mathbf{Z}^{*b}$ mediante la extracción aleatoria de $N$ instancias del conjunto original de árboles $\mathbf{Z}$, utilizando un muestreo con reemplazo. Esto produce que cada muestra de Bootstrap (cada árbol de nuestro modelo) contenga aproximadamente dos tercios de los datos originales, mientras que la cantidad restante se denominan datos \textit{Out-of-Bag} (OOB), los cuales son fundamentales para la validación interna del modelo.

    \vspace{5mm}

    Tras la fase de entrenamiento de cada árbol $T_b$ de cada muestra $\mathbf{Z}^{*b}$, la predicción final del modelo se obtiene comparando los resultados obtenidos de cada árbol. En un problema de clasificación como en nuestro caso, esto se traduce en el voto mayoritario de cada árbol:

    \vspace{2.5mm}

    \begin{equation}
    \hat{f}_{bag}(x) = \frac{1}{B} \sum_{b=1}^{B} T_b(x)
    \end{equation}

    \vspace{2.5mm}

    Donde $\hat{f}_{bag}(x)$ representa la predicción, $B$ el número total de estimadores (árboles) y $T_b(x)$ la salida de cada árbol independiente. De esta manera se reducen los errores de árboles sobreajustados, propiciando en un modelo significativamente más robusto frente al ruido inherente al tráfico de red.

    \vspace{5mm}

    \subsubsection{Feature Selection y Estimación Final}

    \vspace{5mm}

    La principal diferencia entre un \textit{Random Forest} con respecto a un \textit{Bagging} convencional es la segunda capa de aleatoriedad a la hora de construir los distintos árboles. En un modelo \textit{Bagging} se busca la mejor partición entre el conjunto de todas las $p$ variables posibles, mientras que en un \textit{Random Forest} no se tienen en cuenta todas las varibles del conjunto $p$, sino que se selecciona un conjunto aleatorio $m \le p$ de variables, y este número de variables en un problema de clasificación suele ser $m = \sqrt{p}$. 

    \vspace{5mm}

    Esta restricción es lo que nos permite reducir la correlación $\rho$ entre los árboles del bosque. Matemáticamente, la varianza del promedio de $B$ árboles de nuestro modelo, cuya varianza y correlación son $\sigma^2$ y $\rho$ respectivamente, se define como:

    \vspace{2.5mm}

    \begin{equation}
    \text{Var}(\hat{f}_{rf}(x)) = \rho \sigma^2 + \frac{1-\rho}{B} \sigma^2
    \end{equation}

    \vspace{2.5mm}

    Con esto se limitan las variables disponibles para cada nodo, por lo que se reduce el valor de $\rho$ y con ello la varianza total. Esto se conoce como decorrelación y es un proceso que un Bagging simple no puede alcanzar, siendo una de las características principales de un \textit{Random Forest}.
    
    \vspace{5mm}

    Una vez se han entrenados los $B$ árboles del modelo y llega el momento de la predicción (en nuestro caso del tráfico de red en tiempo real) de una nueva observación $x$, entra en juego el estimador. Este estimadorla consolida las predicciones de cada árbol $T_b(x)$ y, en el caso de la clasificación multiclase (DoS, ataques de fuerza bruta, inyecciones SQL, etc.), la predicción final será aquella que se ha repetido un mayor número de veces en las predicciones individuales de los $B$ árboles:

    \vspace{2.5mm}

    \begin{equation}
    \hat{C}_{rf}(x) = \text{mayoría} \{ \hat{C}_b(x) \}_{1}^{B}
    \end{equation}

    \vspace{2.5mm}

    Este asegura que los errores aleatorios de un árbol individual (provocados por ruido o sobreajuste a su muestra de Bootstrap) se compensen con el resto de votos de los árboles restantes, mejorando la robustez del modelo.

    \vspace{10mm}





    \subsection{La Impureza de Gini en un Random Forest}

    \vspace{5mm}

    En esta sección se va a ver una métrica que mide qué tan "mezcladas" se encuentran las distintas clases en un nodo en concreto de un árbol del bosque, la impuereza de Gini. Dado un conjunto de datos con $C$ clases, la impureza de Gini $G$ en un nodo $t$ de un determinado árbol se define como:

    \vspace{2.5mm}

    \begin{equation}
    G(t) = 1 - \sum_{i=1}^{C} (p_i)^2
    \end{equation}

    \vspace{2.5mm}

    Donde $p_i$ es la probabilidad relativa de la clase $i$ en dicho nodo. De esta manera, si $G$ es igual a 0 se dice que dicho nodo es puro. Así mismo, la importancia de una variable se puede calcular mediante la suma de todas las disminuciones de la impureza de Gini que dicha variable produce al ser utilizada para dividir los nodos en todos los árboles del bosque, ponderada por la cantidad de muestras que pasan por dichos nodos. La fórmula para el decremento de impureza $\Delta I$ tras una división $s$ en el nodo $t$ es:

    \vspace{2.5mm}

    \begin{equation}
    \Delta I(s, t) = G(t) - \frac{N_{t_L}}{N_t} G(t_L) - \frac{N_{t_R}}{N_t} G(t_R)  
    \end{equation}

    \vspace{2.5mm}

    Donde $N_{t_L}$ y $N_{t_R}$ son el número de muestras en los nodos hijos, tanto a la izquierda como a la derecha, respectivamente. Entonces, la importancia final de la variable $j$ sería finalmente la media de estas decrementaciones en todos los árboles del bosque. Esto es muy importante, ya que va a ser de gran de ayuda para identificar variables muy importantes para el modelo de clasificación, y otras en cambio, mucho más irrelevantes y que no aportan nada al modelo, incluso siendo negativas en algunos casos al introducir ruido. Esta teoría se implementará más adelante en (\ref{subsubsec: gini_impl}) para concluir con el proceso de \textit{feature selection}.

    \vspace{2.5mm}

    \begin{minted}{python}
    def get_scaled_train_test_data(df_to_scale):
        scaler = StandardScaler()
        X_train, X_test = train_test_split(
            df_to_scale, 
            test_size=0.20, 
            random_state=42
        )
        # Normalizamos los datos para que tengan valores entre 0 y 1
        X_train_scaled = scaler.fit_transform(X_train)
        X_test_scaled = scaler.transform(X_test)
        return X_train_scaled, X_test_scaled
    \end{minted}

    \vspace{2.5mm}

    Con esta función nos estamos ayudando de la librería Scikit-Learn de Python, disponible en \cite{sklearn-docs}. Internamente, se está aplicando la siguiente transformación para el escalado de cada uno de los datos:

    \vspace{2.5mm}

    \begin{equation}
        x_{\text{scaled}} = \frac{x - \mu}{\sigma}
    \end{equation}

    \vspace{2.5mm}

    Donde $\mu$ representa la media aritmética de las muestras de entrenamiento y $\sigma$ la desviación típica. Este proceso se repite para cada columna del dataset, dejando completamente escalados los datos. Con el código anterior también dividimos los datos para el entrenamiento y test, dejando perfectamente preparados los datos para entrenar la red y verificar su correcto funcionamiento, dando paso a la siguiente sección.

    \subsection{Autoencoder}\label{subsec: autoencoder_impl}

    \vspace{5mm}

    Según se vio en (\ref{subsec: autoencoder_subcompleto}), el autoencoder consta de 3 capas: La capa de entrada de datos que aplica una función $f$, obteniendo las características principales para replicar los datos en una capa intermedia, para posteriormente aplicar la función $g$ y replicar los datos de entrada en la capa de reconstrucción o salida de datos. Esto se ve reflejado en el siguiente código:

    \begin{minted}{python}
    def create_and_train_autoencoder(X_train_scaled, X_test_scaled):
        # Creamos la red neuronal (autoencoder)
        input_dim = X_train_scaled.shape[1]
        entrada = Input(shape=(input_dim,))
        encoder = Dense(int(input_dim/2), activation='relu')(entrada)
        decoder = Dense(input_dim, activation='linear')(encoder)

        autoencoder = Model(inputs=entrada, outputs=decoder)
        autoencoder.compile(optimizer='adam', loss='mse')

        print('Comenzamos el entrenamiento de la red...')
        history = autoencoder.fit(
            X_train_scaled,
            X_train_scaled,
            epochs=20,
            batch_size=256,
            validation_data=(X_test_scaled, X_test_scaled),
            callbacks=[early_stop],
            verbose=1
        )
        return autoencoder, history
    \end{minted}   
    
    Donde vemos que las funciones de activación $f$ y $g$ son las funciones \textit{ReLu} y la función lineal respectivamente. En primer lugar se usa la función \textbf{ReLu} ya que es la función por excelencia en capas ocultas al solucionar el desvanecimiento de gradiente, y al ser lineal en los valores positivos permite a la red aprender muy rápido teniendo en cuenta las características más relevantes del tráfico de red, debido a que los valores negativos son cero.

    \vspace{2.5mm}

    \begin{equation}
        f(x) = \text{max}\left(0, x\right)
        \label{eq: relu}
    \end{equation}


    \vspace{2.5mm}

    Por otra parte, se ha optado por una función de activación lineal para la capa de salida. Normalmente se usaría la función Sigmoidea ya que la salida de las redes suele estar normalizada. Sin embargo, debido a la alta presencia de valores atípicos, este no es nuestro caso. Dado que nuestros datos han sido estandarizados mediante un \textit{StandardScaler}, los valores de la capa de entrada, y por lo tanto también los de reconstrucción, ya no se encuentran sujetos a escala $[0, 1]$, sino que pueden alcanzar valores significativamente superiores al toparse con cualquier anomalía. De esta manera una función lineal permite que las neuronas de salida repliquen cualquier valor sin sufrir el efecto de saturación que provocarían funciones como la sigmoide, lo que resulta fundamental para que el error cuadrático medio (MSE) refleje con precisión la distancia estadística entre el tráfico benigno y un ataque malicioso masivo.

    \vspace{2.5mm}

    \begin{equation}

        \label{eq: lineal}
    \end{equation}

    \subsection{Determinación del Umbral de Detección de Anomalias}\label{subsec: determinacion_umbral_deteccion_anomalias}

    \vspace{5mm}

    Una vez ya se han realizado las pruebas de penetración anteriores y el modelo responde correctamente, se ha de determinar el umbral. Este umbral $(\tau)$ es la frontera que separa el tráfico benigno del malicioso, lo que en la práctica se traduce como el valor mínimo de MSE para detectar tráfico con intenciones maliciosas. Lo correcto es que este valor no sea escogido a la ligera, sino que se analicen distintos métodos para su obtención, comprobando el mecanismo más robusto para su implementación en nuestro sistema de detección de intrusiones en tiempo real.

    \vspace{5mm}

    En primera instancia, se tomaron valores reales de tráfico de paquetes en una red doméstica. Se pudo haber usado directamente los datos de entrenamiento, pero consideramos que esto es una mejor opción debido a la variabilidad de un entorno real, que frente a la naturaleza estadística de nuestros datos de entrenamiento, incluyen ruido del sistema operativo, latencia de red, diferencias en las cabeceras HTTP, etc. Es por esto que se almacenaron datos del MSE como resultado de pasar dichos flujos de red por el autoencoder, flujos realizados desde distintos dispositivos y cuyo archivo se encuentra en el repositorio de Github de este trabajo \cite{github_tfg}.

    \vspace{5mm}

    Por ello, se implementaron tres distintas metodologías con el fin de que dicho umbral sea lo más robusto posible, minimizando falsos positivos pero sin llegar a comprometer el sistema ante ciberataques más sigilosos (como la prueba realizada con Hydra anteriormente en (\ref{fig: mse_hydra})).

    \vspace{10mm}

    \subsubsection{Método de la Desviación Estándar}\label{subsubsec: metodo_desviacion_estandar}

    \vspace{5mm}

    En este método asumimos que el MSE sigue una distribución normal. Entonces, podemos definir el umbral en función de la media ($\mu$) y la desviación estándar ($\sigma$):

    \vspace{2.5mm}

    \begin{equation}
        \tau = \mu + \left( k \cdot \sigma \right)
    \end{equation}

    \vspace{2.5mm}

    Donde $k$ representa un factor de sensibilidad. En función de este valor se puede ser más conservador y abarcar prácticamente todo el tráfico benigno en su totalidad, o se puede tomar un enfoque más radical. No obstante, este último enfoque no es el más adecuado ya que generaría muchos falsos positivos. Inicialmente se propuso un valor $k = 3$ para las primeras pruebas, aunque más adelante se modificó este valor hasta $k = 5$ para garantizar un entorno altamente conservador. Con este dato y conociendo el valor de $\mu$ y $\sigma$ se puede hallar el umbral en función de la desviación estándar como sigue:

    \vspace{2.5mm}

    \begin{equation*}
        \tau = \mu + \left( k \cdot \sigma \right) = 1.4366 + \left( 5 \cdot 1.5654 \right) = 9.2634
    \end{equation*}

    \vspace{2.5mm}

    Con este método logramos obtener un umbral de $\tau = 9.2634$, el cual veremos que es el más conservador de los tres modelos de umbrales que se va a ver.

    \vspace{10mm}

    \subsubsection{Método de la Percentiles}\label{subsubsec: metodo_percentiles}

    \vspace{5mm}

    Este segundo enfoque, a diferencia del modelo anterior (\ref{subsubsec: metodo_desviacion_estandar}), no asume una distribución en específico, sino que se centra en la estadística, en el percentil $P$, lo que indica que el $P\%$ de los datos se encuentran por debajo del umbral, como sigue:

    \vspace{2.5mm}

    \begin{equation}
        \tau = P_{99.9} \left( \text{MSE}_{benigno} \right) = 6.5387
    \end{equation}

    \vspace{2.5mm}

    Sin embargo, como se puede observar, este valor es inferior al anterior, y debido al enfoque conservador que se quiere emplear en este apartado, se ha decidido finalmente por descartar este método, para disminuir al máximo la posibilidad de obtener falsos positivos al analizar el flujo de red.

    \vspace{10mm}

    \subsubsection{Método del Valor Máximo}\label{subsubsec: metodo_valor_maximo}

    \vspace{5mm}

    Esta última metodología consiste en identificar el valor máximo de los datos de prueba capturados previamente, añadiendo un margen de guarda para establecer el umbral:

    \vspace{2.5mm}

    \begin{equation}
        \tau = \text{max}\left( MSE_{benigno} \right) + \Delta
    \end{equation}

    \vspace{2.5mm}

    Sin embargo este método tampoco terminó de encajar, ya que depende mucho del margen de guarda que se escoja. Es por esto que se decidió implementar el método de la desviación estándar (\ref{subsubsec: metodo_desviacion_estandar}) con un valor de $k = 5$, ya que a diferencia de este último método, se tiene en cuenta la distribución de los datos, lo que permite ignorar picos de ruido de forma puntual mientras se mantiene la capacidad de detección de ciberataques como las pruebas de penetración vistas anteriormente en (\ref{subsec: pruebas_penetracion_sqlmap_hydra}), manteniendo un enfoque conservador y disminuyendo los falsos positivos.

    \vspace{2.5mm}

    \begin{equation*}
        \boxed{\tau = 9.2634}
    \end{equation*}

    \subsubsection{Primeros resultados}\label{subsubsec: primeros_resultados}

    \vspace{5mm}

    Tras un primer entrenamiento del \textit{Random Forest}, lo resultados no coincidieron con lo esperado, y esto se debe a la naturaleza de los datos de partida. Tras un previo análisis, se identificó un fuerte sesgo en la distribución de muestras de carácter malicioso, ya que ciertos ataques, \textit{DoS Hulk} y \textit{DoS Goldeneye} para ser más específicos, representan una gran descompensación con el resto de amenazas:

    \vspace{5mm}

    \begin{figure}[H]
        \centering
        \includegraphics[width=1\textwidth]{../img/code/class_distribution_malign_cleaned.png}\\
        \caption{Distribución de clases de carácter malicioso tras análisis de duplicados}
        \label{}
    \end{figure}  

    \vspace{5mm}

    Este gráfico muestra el sesgo que se ha hablado anteriormente, mostrando que tras un análisis de datos duplicados la cantidad de datos maliciosos se concentra mucho más entorno a \textit{DoS Hulk} y \textit{DoS Goldeneye}, lo que hace que el resto de opciones sean de carácter invisible en la clasificación al no presentar apenas datos. Este desequilibrio conlleva una serie de problemas en la clasificación del modelo:

    \vspace{5mm}

    \begin{itemize}
        \item \textbf{Sesgo Algorítmico:} Al presentar una desproporción tan grande de datos en las muestras de \textit{Hulk} y \textit{GoldenEye}, los distintos clasificadores del bosque aprenden con mayor detalle este tipo de tráfico pero quedan estancados en un aprendizaje mucho más superficial y general con el resto de ataques.
        
        \vspace{2.5mm}

        \item \textbf{Predominancia de Falsos Positivos:} De manera ligada al punto anterior, al conocer mucho mejor las características de un par de clasificaciones en concreto, el modelo genera una fuerte tendencia a clasificar las opciones que mejor conoce, las cuales son \textit{DoS Hulk} y \textit{DoS Goldeneye}, por una mera cuestión de probabilidad derivada en el entrenamiento.
        
        \vspace{2.5mm}

        \item \textbf{Generalización Deficiente:} Al presentar una gran escasez de muestras en ataques como \textit{SQL Injection} o \textit{Botnets} impiden que el algoritmo identifique patrones muy característicos, generando muy poca confianza en la predicción del resto de clases.
    \end{itemize}

    \vspace{5mm}

    Tras esto se puede explicar el resultado de innumerables pruebas erróneas generadas por el modelo, reproduciendo ataques con distintas herramientas presentadas en (\ref{subsec: pruebas_penetracion_sqlmap_hydra}) y observando que los flujos de red generados son etiquetados incorrectamente como \textit{Dos GoldenEye} o \textit{DoS Hulk} en la gran mayoría de los casos. Además, sumado a una baja confianza del modelo a la hora de la clasificación final  (normalmente cercana al 40\%) quiere decir que el modelo se encuentra con mucha indecisión debido a este fuerte sesgo.

En la siguiente sección se muestra la medida adoptada para solucionar este problema, mediante la generación de tráfico  adicional, implementado directamente en un entorno de red propio y mucho más controlado, con total transparencia a la hora de mostrar las herramientas y metodologías implementadas para mitigar este problema y mejorar la equidad en la detección de todas las opciones posibles.

    \subsubsection{Captura de datos personalizada}\label{subsubsec: captura_datos_personalizada}

    \vspace{5mm}

    Una vez se ha detectado el sesgo visto en la sección anterior (\ref{subsubsec: primeros_resultados}), se ha capturado el tráfico malicioso de la reproducción de ataques en una red doméstica, simulando un entorno de laboratorio controlado para la captura. Para la simulación de ataques de denegación de servicio \textit{DoS} se ha utilizado la herramienta \texttt{hping3}, la cual permite generar fuertes inundaciones de paquetes TCP/IP, al igual que \texttt{sqlmap} para inyecciones \textit{SQL}, ya que automatiza la detección de vulnerabildades \textit{SQL}.

    \vspace{5mm}

    Por otra parte, nos hemos apoyado en la herramienta de código abierto creada por Janse de Visser y Jan Seidl \cite{goldeneye_repo} para ataques \textit{DoS Goldeneye}, la cual se puede ejecutar a través de un fichero de Python, de la misma manera que uno de los scripts creados para este proyecto, \texttt{bruteforce.py}, el cual se puede encontrar en el repositorio de este trabajo \cite{github_tfg}, generando un ataque de fuerza bruta (\textit{Bruteforce}), al igual que se ha hecho con \texttt{hydra} como se vio anteriormente en (\ref{fig: mse_hydra}).

    \vspace{5mm}

    Por último, ya que no se han implementado todos los ataques que se disponen en los datos originales, se ha concluido con \texttt{slowhttptest}, especializándose en ataques de bajo ancho de banda como \textit{DoS Slowhttptest}. Con estos datos se puede entrenar correctamente un modelo mucho más homogéneo en cuanto a distribución de clases, propiciando un buen desempeño del modelo con predicciones mucho más precisas, ya que el flujo generado para el entrenamiento tiene características mucho más similares a los datos del entorno de pruebas, erradicando el sesgo que provocaba fuertes confusiones en la clasificación final:

    \vspace{5mm}

        \label{}
    \end{figure}  

    \vspace{5mm}

    Finalmene, esta captura personalizada de datos resultó ser fundamental para el equilibrio de los datos, ya que permite que el clasificador \textit{Random Forest} sea entrenado con una representación más equitativa y realista de cada categoría de ataque, lo que se traduce en unos resultados mucho más acorde a lo esperado, como se verá en la siguiente sección.

    \subsubsection{Resultados finales}

    \vspace{5mm}

    Antes de mostrar las principales métricas como la \texttt{precision}, \texttt{accuracy}, \texttt{recall} o \texttt{score f1}, se van a analizar de manera detallada las muestras generadas ante el desequilibrio de datos que se vio en \ref{subsubsec: primeros_resultados}. Vamos a tomar como ejemplo las inyecciones SQL, tomadas a través de la herramienta \texttt{sqlmap}:

    \vspace{5mm}

    \begin{figure}[H]
        \centering
        \includegraphics[width=1\textwidth]{../img/code/df/sql_injection_samples.png}\\
        \caption{Muestras de datos capturados: SQL Injection}
        \label{}
    \end{figure} 

    \vspace{5mm}
    
    Esto revela una homogeneidad muy alta en los datos tomados, ya que se puede identificar muy fácilmente la presencia de patrones que determinan la estructura de este tipo de ataque en concreto. En particular, se pueden encontrar los siguientes patrones en flujos generados por la herramienta \texttt{sqlmap}:

    \vspace{5mm}

    \begin{itemize}
        \item \textbf{Simetría de Flujo:} El ratio de subida/bajada (\texttt{Down/Up Ratio}) almacena un valor constante de 1.0, al igual que la cantidad total de paquetes enviados  (\texttt{Total Fwd Packets}) que también presenta el mismo valor, en este caso 5 paquetes por flujo.
        
        \vspace{2.5mm}

        \item \textbf{Flags TCP:} También se puede observar que las banderas de control TCP (\texttt{FIN}, \texttt{PSH}, \texttt{ACK}) presentan el mismo patrón, activándose exactamente las mismas banderas en cada flujo de red generado.
        
        \vspace{2.5mm}

        \item \textbf{Límites de Carga:} Por último se aprecia que las longitudes mínimas de los paquetes (\texttt{Min Packet Length}, \texttt{Fwd Packet Length Min}, \texttt{Bwd Packet Length Min}) presentan un valor homogéneo y constante en todos los flujos capturados, estabilizándose en un total de 68 bytes.
    \end{itemize}

    \vspace{5mm}
 
    Esto provoca que, durante la fase de evaluación del modelo, se obtengan métricas casi perfectas, alcanzando valores de $0.99$ sobre $1$ en \textit{Accuracy}, \textit{Precision}, \textit{Recall} y \textit{F1-Score} como consecuencia de la alta homogeneidad de los datos capturados:

    \vspace{5mm}

    \begin{figure}[H]
        \centering

    \end{figure} 

    \vspace{5mm}

    Desde una perspectiva analítica, esto justifica matemáticamente las métricas que se han obtenido. Sin embargo, esto salta las alarmas, ya que es un claro ejemplo de \textbf{sobreentrenamiento (overfitting)}, ante la grave dificultad para detectar un ataque de forma genérica, teniendo el modelo únicamente la capacidad de detectar la huella de red generada por ciertas herramientas, como \texttt{sqlmap}, bajo una configuración en específico en el entorno de pruebas. Si se hace la prueba, se puede verificar que se tiene mucha confianza en las predicciones, como resultado de los fenómenos recientemente explicados:

    \vspace{5mm}

    \begin{figure}[H]
        \centering
        \includegraphics[width=0.75\textwidth]{../img/mse/sqlmap_attack.png}\\
        \caption{Prediciendo ataque malicioso creado con SQLMap}

    \vspace{5mm}

    En conclusión, aunque esto pueda aparentar inicialmente un gran éxito del clasificador, el análisis pormenorizado de los flujos capturados sugiere una realidad más compleja. La presencia de patrones deterministas (firmas de red como las flags y métricas de volumen y longitud de datos) indica que el modelo converge a una solución muy óptima para  nuestro entorno controlado, pero a la vez es potencialmente frágil ante un escenario real. Este fenómeno recalca la importancia de la diversidad en los datos, ya que se demuestra una precisión excepcional para identificar ataques producidos por herramientas en concreto, con el coste de perder la capacidad de generalizar mucho más allá y lograr adaptarse a la enorme variabilidad intrínseca del tráfico malicioso.

    \subsection{Implementación de Notificaciones y Respuesta ante Incidentes vía Telegram API}

    \vspace{5mm}

    A continuación se ha decidido implementar la API de Telegram en este proyecto, lo que proporciona un sistema de respuesta y monitarización remota a través de un bot. Esta implementación permite establecer un canal de comunicación entre la detección del sistema y su propio administrador, lo que facilita la toma de decisiones frente a posibles intrusiones en tiempo real.

    \vspace{5mm}

    La notificación se produce inmediatamente después de la clasificación del \textit{Random Forest}, en caso de que se haya detectado tráfico malicioso, en caso contrario, no se envía ningún mensaje. Esta comunicación se establece mediante peticiones HTTP POST al \textit{endpoint} de la API de Telegram, y es muy importante recalcar que la comunicación es bidereccional, de manera que al detectarse un posible ataque el administrador puede decidir si bloquear la dirección IP o no, como se puede ver a continuación:

    \vspace{5mm}

    \begin{figure}[H]

En caso contrario, se continúa con la detección de anomalías. Esta es la cúlpide del sistema, ya que permite notificar al administrador del servidor de un posible ataque, ofreciendo las medidas necesarias para hacer frente a este tipo de amenazas y mitigando los daños que puedan causar.
